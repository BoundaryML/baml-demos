# Google ADK + BAML Demo

A small **expense-approver agent** built with the [Google Agent Development Kit](https://google.github.io/adk-docs/), whose tools are **[BAML](https://docs.boundaryml.com) functions** called through the generated Python SDK.

The split is the point of the demo:

- **Gemini (via ADK)** owns the conversation — it decides *when* to call which tool and writes the final answer.
- **BAML** owns *what the tools do* — one tool is a typed LLM extraction function, the other is deterministic typed business logic with no LLM in it at all.

```
user ──► ADK Agent (gemini-2.5-flash)
              │ tool call                      │ tool call
              ▼                                ▼
        parse_expense(text)             check_expense_policy(amount, currency, category)
              │ baml_sdk                       │ baml_sdk
              ▼                                ▼
        BAML LLM function               BAML pure function
        (gpt-4o-mini, schema-          (match on Category, exact
         validated Expense)             limits, testable offline)
```

## The two tools

| Tool | BAML source | What it is |
|---|---|---|
| `parse_expense` | `baml_src/expense.baml` | LLM function: messy note (`"ubr 23.50 to airport tue"`) → typed `Expense`. The return type *is* the output schema — BAML injects it via `ctx.output_format`, validates the reply, and re-prompts on malformed JSON. |
| `check_expense_policy` | `baml_src/policy.baml` | Ordinary typed BAML — per-category limits, receipt threshold, non-USD flagging. No model call, so decisions are exact, auditable, and unit-tested offline. The agent is instructed to use it instead of guessing policy from its weights. |

The ADK-side wrappers in `expense_agent/agent.py` are one line of logic each: call the generated `baml_sdk` function, `model_dump()` the Pydantic result into the dict the ADK expects. Docstrings + type hints on the wrappers are what Gemini sees as the tool schema.

## Layout

```
google-adk/
├── baml.toml                # package + [generator.app] → python/pydantic
├── baml_src/
│   ├── clients.baml         # ExpenseModel (OpenAI gpt-4o-mini)
│   ├── types.baml           # Category, Expense, PolicyDecision
│   ├── expense.baml         # parse_expense — the LLM tool
│   ├── policy.baml          # check_policy — the deterministic tool
│   └── policy_test.baml     # offline tests, no API keys needed
├── baml_sdk/                # generated by `baml generate` (gitignored)
├── expense_agent/
│   ├── __init__.py
│   └── agent.py             # root_agent + tool wrappers over baml_sdk
├── run_agent.py             # one-shot programmatic run, prints tool calls
└── pyproject.toml
```

## Setup

```bash
brew install boundaryml/tap/baml                 # BAML wrapper
baml toolchain use canary                        # BAML CLI
uv sync                                          # Python deps (google-adk, baml_core, …)
baml generate                                    # emit ./baml_sdk/
```

Two keys for a live run — the agent and the tools deliberately run on different providers:

```bash
# .env  (loaded by run_agent.py and by `adk run` / `adk web`)
GOOGLE_GENAI_USE_VERTEXAI=FALSE
GOOGLE_API_KEY=...        # the ADK agent (Gemini)
OPENAI_API_KEY=...        # the BAML parse_expense tool
```

## Run

**One-shot script** — prints every tool call and typed result as the loop runs:

```bash
uv run python run_agent.py
uv run python run_agent.py "team lunch 43.20 @ chipotle, need this back"
```

**Interactive** — the standard ADK entry points, from this directory:

```bash
uv run adk run expense_agent    # terminal chat
uv run adk web                  # dev UI at localhost:8000
```

## Test without any keys

The policy tool is plain BAML, and `parse_expense`'s *parsing* path is exercised against a captured model reply via the compiler-synthesized `parse_expense$parse` — so the test suite is fully offline:

```bash
baml test
# 6 passed, 0 failed
```

You can also poke the deterministic tool directly:

```bash
baml run -e 'check_policy(184.0, "USD", Category.Meals)' --output-format json
# {"status":"needs_approval","limit":75.0,"over_by":109.0,"reasons":["exceeds the 75.0 USD category limit"]}
```

## Why bother routing tools through BAML?

- **Schema enforcement where it matters.** ADK gets a plain dict back, but *inside* the tool the extraction is validated against `Expense` and retried on garbage — the agent never sees a half-parsed record.
- **Policy is code, not vibes.** `check_policy` is a `match` over an enum with exact float math. `baml test` proves it offline; no prompt drift can change a limit.
- **Types cross the boundary.** `baml generate` gives Pydantic models, so `expense.model_dump(mode="json")` is the whole marshalling story.
- **Providers stay decoupled.** Gemini orchestrates while the tools run OpenAI; either side can be swapped in one line (`model=` in `agent.py`, `client:` in `clients.baml`).
